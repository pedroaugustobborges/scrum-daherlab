<h1>API Rate Limits</h1>
<p>The API has rate limits in place to ensure fair usage and prevent abuse.
These limits are designed to protect both the API users and the API itself from excessive traffic.</p>
<p>There are two rules:</p>
<ol>
<li>Burst Protection allows a maximum of 100 messages per second per user.</li>
<li>API Usage limit allowing a maximum of 1000 messages per minute per user.</li>
</ol>
<p>For higher-volume needs, get in touch so we can evaluate your use case and potential options.</p>
<h1>Idempotency</h1>
<p>The API supports idempotency for safely retrying requests without accidentally performing the same operation twice.</p>
<p>To perform an idempotent request, provide an additional <code>Idempotency-Key: &lt;key&gt;</code> header to the request.</p>
<p>The idempotency key should be a unique ULID value generated by the client.</p>
<h2>How it works</h2>
<ol>
<li>Send a request with an <code>Idempotency-Key</code> header</li>
<li>The server stores the response for that idempotency key</li>
<li>If you retry the request with the same key, the server returns the stored response</li>
<li>The idempotency key is valid for 24 hours</li>
</ol>
<h2>Error handling</h2>
<ul>
<li>If you send a request without an idempotency key or with an invalid idempotency key format, you'll receive a 400 Bad Request error</li>
<li>If you retry a request while the original request is still processing, you'll receive a 409 Conflict error</li>
</ul></div></div></div>
